### Découverte importante (audit v1 erroné)

**8 des 12 bugs critiques signalés au v1 n'existaient pas.** L'audit v1 testait au curl avec des payloads malformés ou de mauvaises URLs. Vérifié en direct API au syndic réel :

- `POST /meetings`, `/expenses`, `/polls`, `/board-members`, `/convocations`, `/journal-entries`, `/etats-dates`, `/organizations` retournent **201/200** avec un payload valide construit par le frontend.
- Le **SEL n'est pas absent** : `POST /exchanges` et `GET /buildings/{id}/exchanges` fonctionnent. Le `/api/v1/exchanges` (GET sans suffixe) est volontairement absent — c'est l'audit qui s'était trompé d'URL.
- Les **routes admin** sont `/organizations` et `/users` (sans préfixe `/admin/`). L'audit v1 cherchait `/admin/organizations` qui n'existe pas.
- Le **login autocomplete** est correct (`autocomplete="email"` / `current-password`) — la suggestion v1 de mettre `autocomplete="off"` casserait les password managers.

## Méthodologie OBLIGATOIRE

**Règle 1** : tester via l'UI, pas au curl. Les payloads attendus sont définis par les composants frontend, c'est ce comportement qu'il faut tester.

**Règle 2** : pour tout bug "POST 4XX", **screenshot DevTools Network obligatoire** avec le payload envoyé et la réponse reçue. Sans ça, le bug n'est pas accepté dans le rapport.

**Règle 3** : avant de déclarer "endpoint absent", chercher dans `backend/src/infrastructure/web/routes.rs` et les handlers correspondants. Beaucoup d'endpoints sont nichés sous `/buildings/{id}/...` ou `/owners/{id}/...` plutôt que des collections plates.

**Règle 4** : seed propre. Re-seed avant de commencer :
```bash
TOKEN=$(curl -s http://localhost/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@koprogo.com","password":"admin123"}' | jq -r .token)
# Si le scenario existe déjà, le supprimer d'abord :
curl -X DELETE http://localhost/api/v1/seed/scenario/world -H "Authorization: Bearer $TOKEN"
curl -X POST http://localhost/api/v1/seed/scenario/world -H "Authorization: Bearer $TOKEN"
```

**Règle 5** : login rate-limité (5 essais / 15min / IP). Privilégier `victor@terrasses-flagey.be` / `victor123` qui n'a pas été abusé pendant les v1–v5.
