## Les 10 parcours multi-rôles à tester

Pour chaque parcours : suivre l'ordre des étapes, se reconnecter avec le bon rôle à chaque changement, prendre une capture AVANT et APRÈS chaque action critique, observer la cohérence des données à travers les rôles.

### P1 — Cycle de vie d'un ticket de maintenance
**Acteurs** : Charlie (owner) → François (syndic)

1. **Charlie** se connecte, va sur `/tickets`, crée un ticket "Fuite robinet cuisine lot 3B" priorité High
2. Vérifier : le ticket apparaît avec statut `Open` et due_date à 24h
3. **François** se connecte, voit le ticket sur son dashboard
4. François assigne le ticket à un contractor
5. Vérifier : statut passe à `Assigned`, Charlie reçoit-il une notification in-app ?
6. François marque `start`, puis `resolve` avec notes
7. Charlie voit-il le statut `Resolved` ? Peut-il commenter / rouvrir ?

**À observer** : cohérence statuts, notifications, permissions (un owner peut-il modifier un ticket qui n'est pas le sien ?), due_date calculation

---

### P2 — Vote sur résolution en AG (multi-rôles)
**Acteurs** : François (syndic) → Alice, Bob, Charlie (owners)

1. **François** crée une AG : `Meetings > New`, date, location, meeting_type "Ordinary"
2. François crée une résolution "Approbation du budget 2026" (majorité absolue)
3. François publie/envoie convocations (vérifier délai légal 15j Art. 3.87 §3 CC)
4. **Alice** se connecte, va sur `/meetings/:id`, vote `Pour` avec son voting_power (450 tantièmes)
5. **Bob** vote `Contre` (430 tantièmes)
6. **Charlie** vote `Abstention` (660 tantièmes)
7. François clôture le scrutin : le résultat est-il calculé correctement ? (majorité absolue = 50%+1 des votes exprimés, abstentions exclues)
8. Vérifier : procès-verbal généré, visible par tous les owners

**À observer** : calcul de majorité, gestion des abstentions, timestamps, proxy (mandataire), visibilité PV

---

### P3 — Workflow d'approbation de facture (Draft→PendingApproval→Approved→Paid)
**Acteurs** : François (syndic) → Gisèle (comptable)

1. **François** va sur `/expenses`, crée une facture "Nettoyage parties communes" 300€ HTVA 6%
2. Upload d'un justificatif (PDF ou image)
3. François soumet pour approbation
4. **Gisèle** se connecte, voit la facture dans sa liste d'approbations
5. Gisèle approuve
6. Vérifier : le statut passe à `Approved`, la répartition des charges est calculée automatiquement sur les 10 lots selon tantièmes
7. Chaque owner voit-il sa quote-part dans son dashboard `/owner/expenses` ?

**À observer** : calcul VAT belge (6%/12%/21%), répartition tantièmes, permissions Gisèle vs François, upload PDF

### P4 — Sondage rapide entre AG (Polls — Art. 577-8/4 §4 CC)
**Acteurs** : François (syndic) → Alice, Bob (owners)

1. **François** crée un sondage YesNo "Repeindre le hall d'entrée en bleu ?" avec 2 options Oui/Non
2. François publie (Draft → Active)
3. **Alice** vote Oui, **Bob** vote Non
4. François clôture et consulte les résultats
5. Vérifier : graphique de résultats, participation rate, pas de possibilité de revote

**À observer** : anonymous vs non-anonymous, allow_multiple_votes=false, graphiques, export résultats

### P5 — SEL (Système d'Échange Local)

4. Vérifier : le solde d'Alice augmente de 2 crédits, celui de Bob diminue de 2
5. Alice note Bob, Bob note Alice (1-5 étoiles)
6. Vérifier : leaderboard du bâtiment, statistiques SEL

**À observer** : solde négatif autorisé, gamification (participation level), notation mutuelle

## Consignes strictes

1. **Prends toujours une capture avant ET après** chaque action importante (click sur bouton d'action, soumission de formulaire, changement de statut)
2. **Nomme les captures** : `p{N}-step{M}-{role}.png` (ex : `p2-step4-alice.png`)
3. **Logout+login explicite** à chaque changement de rôle (pas de switch silencieux)
4. **Vérifie les permissions** : essaie délibérément des actions non autorisées et documente la réponse (403 ? redirect ? message d'erreur ?)
5. **Note les temps de réponse** anormaux (>2s perçu)
6. **Vérifie la cohérence des données** à travers les rôles : ce que François voit doit être cohérent avec ce que Alice voit
7. **Test des langues** : bascule FR → NL → EN → DE sur quelques écrans clés, note les clés i18n manquantes
8. **Accessibility quick check** : navigation clavier, contrastes évidents, alt text sur images
9. **Ne corrige rien** : tu documentes, tu ne fixes pas le code
10. **En cas de crash** : capture l'erreur + log console + network tab
