## Ce qui a changé depuis le v2

### Fix #v2-1 — `building_id` manquant dans POST `/tickets` (`TicketCreateModal.svelte`)

**Cause racine identifiée** : sous Svelte 5 `mount()`, l'instruction `let formData = { building_id: buildingId }` capture le prop `buildingId` AVANT qu'il ne soit assigné au composant. Résultat : `formData.building_id = ""` même quand le prop est passé correctement. Le composant `tickets.astro` mountait bien la modale avec `props: { buildingId: "<uuid>" }`, mais l'init du formData captait la valeur par défaut `""`.

**Fix** : double sécurité dans `handleSubmit()` :
```typescript
if (!formData.building_id && buildingId) {
  formData.building_id = buildingId;
}
// Et dans le payload :
ticketsApi.create({
  ...formData,
  building_id: formData.building_id || buildingId,  // fallback prop
  ...
})
```

### Fix #v2-2 — Footer modale ticket non sticky (`TicketCreateModal.svelte`)

**Cause racine** : le composant `Modal` partagé utilise `<slot name="footer" />` avec `{#if $$slots.footer}`. Cette détection conditionnelle de slot ne fonctionne **pas** de manière fiable avec Svelte 5 `mount()` — les boutons que j'avais mis dans `<div slot="footer">` n'étaient jamais détectés et tombaient dans le slot par défaut, donc dans la zone scrollable.

**Fix** : abandon du slot nommé. Les boutons sont maintenant dans un `<div class="sticky bottom-0 bg-gray-50 border-t border-gray-200 p-4 ...">` à l'intérieur du formulaire. Le `form` a `class="-m-6"` pour annuler le padding du Modal et le contenu est wrappé dans `<div class="space-y-4 p-6 pb-4">`. Le `sticky bottom-0` colle les boutons en bas de la zone de scroll quel que soit le wrapping parent.

### Fix #v2-3 — Statistiques tickets vides (backend serde rename)

**Cause racine identifiée — c'était un mismatch de schéma JSON**, pas un bug de refresh ni de pagination. Le backend retournait :
```json
{ "total": 1, "open": 1, "in_progress": 0, ... }
```
alors que le composant frontend `TicketStatistics.svelte` lit :
```typescript
stats.total_tickets, stats.open_tickets, stats.in_progress_tickets, ...
```
→ tous `undefined` → toutes les cartes affichaient leur label sans valeur.

**Fix** : ajout de `#[serde(rename = "total_tickets")]` etc. sur le struct backend `TicketStatistics`. Ajout des champs manquants `assigned_tickets` (toujours 0 — le domaine Rust n'a pas d'état "Assigned" séparé, c'est seulement frontend) et `overdue_tickets` (calculé via `ticket.is_overdue(7)`).

Le nouveau payload retourné :
```json
{
  "total_tickets": 1,
  "open_tickets": 1,
  "assigned_tickets": 0,
  "in_progress_tickets": 0,
  "resolved_tickets": 0,
  "closed_tickets": 0,
  "cancelled_tickets": 0,
  "overdue_tickets": 0
}
```

### Fix #v2-4 — Statuts non traduits dans la liste (`TicketStatusBadge.svelte`)

**Cause** : les labels étaient hard-codés en anglais dans le composant badge. Les clés i18n existent depuis le départ (`tickets.statuses.open` = "Ouvert", etc.) mais n'étaient pas utilisées.

**Fix** : utilisation de `$_(config.key)` au lieu de `config.label`.

⚠️ **Attention au mapping** : la clé pour `InProgress` est `tickets.statuses.inProgress` (camelCase), pas `in_progress` ni `inprogress`. Vérifié dans `frontend/src/locales/fr.json`.
