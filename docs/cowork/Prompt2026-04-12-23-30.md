## Ce qui a changé depuis le v4

### Fix #v4-1 — Alignement enum catégorie ticket frontend ↔ backend

**Cause racine** : le backend Rust expose 9 catégories (`Plumbing`, `Electrical`, `Heating`, `CommonAreas`, `Elevator`, `Security`, `Cleaning`, `Landscaping`, `Other`) mais le frontend en exposait 7 différentes dont 2 inventées (`General`, `Emergency`) qui causaient `400 unknown variant General`. La catégorie par défaut était `General` → tous les utilisateurs étaient bloqués.

**Vérifié au backend** : POST /api/v1/tickets avec chacune des 9 catégories backend → **201 Created** pour les 9. Le mismatch était purement frontend.

**Fix** : alignement complet du frontend sur l'enum backend.
- [`frontend/src/lib/api/tickets.ts`](frontend/src/lib/api/tickets.ts) — enum `TicketCategory` réécrit avec les 9 valeurs Rust
- [`frontend/src/components/tickets/TicketCreateModal.svelte`](frontend/src/components/tickets/TicketCreateModal.svelte) :
  - default `category: TicketCategory.Other` (au lieu de `General`)
  - reset après close idem
  - dropdown avec les 9 nouvelles options
- [`frontend/src/components/tickets/TicketList.svelte`](frontend/src/components/tickets/TicketList.svelte) — filtre catégorie aligné sur les 9 valeurs
- [`frontend/src/components/tickets/TicketDetail.svelte`](frontend/src/components/tickets/TicketDetail.svelte) et [`TicketList.svelte`](frontend/src/components/tickets/TicketList.svelte) — la conversion `category.toLowerCase()` pour le lookup i18n a été remplacée par `category.charAt(0).toLowerCase() + category.slice(1)` (PascalCase → camelCase) pour matcher les clés JSON. `"CommonAreas"` → `"commonAreas"`, pas `"commonareas"`.
- 4 fichiers de locale mis à jour (`fr.json`, `nl.json`, `de.json`, `en.json`) :
  - retrait des clés `general` et `emergency` (orphelines)
  - ajout de `commonAreas`, `elevator`, `landscaping`, `other`
