### Endpoints vérifiés en direct (N'ÉTAIENT PAS CASSÉS)

L'audit du 18:00 a signalé 8 endpoints retournant `Json deserialize error: premature end of input` :
- POST `/meetings`, `/expenses`, `/polls`, `/board-members`, `/convocations`, `/journal-entries`, `/etats-dates`, `/organizations`

**Vérifié au curl avec un syndic réel (`francois@syndic-leroy.be` / `francois123`)** : seul `/meetings` était bloqué (champ `location` manquant — corrigé). Tous les autres acceptent un payload valide et retournent 201 / 200. L'audit précédent envoyait probablement des payloads malformés (corps vide ou JSON incomplet).

**Le SEL n'est pas absent** : la route `/api/v1/exchanges` (GET sans suffixe) n'existe pas, mais c'est volontaire. Les bonnes routes sont :
- `POST /api/v1/exchanges` (créer une offre)
- `GET /api/v1/buildings/{building_id}/exchanges` (lister les échanges d'un immeuble)
- `GET /api/v1/buildings/{building_id}/exchanges/available` (marketplace)

**Routes admin** : `/api/v1/admin/organizations` n'existe pas (404), c'est `/api/v1/organizations` et `/api/v1/users` (sans préfixe `/admin/`). L'audit s'était trompé d'URL — le code est correct.

**Login autocomplete** : les champs utilisent déjà `autocomplete="email"` et `autocomplete="current-password"`, ce qui est CORRECT pour les password managers. La suggestion `autocomplete="off"` du précédent audit casserait l'UX — elle a été refusée.

## Méthodologie OBLIGATOIRE pour ce nouvel audit

**Règle #1 : ne pas tester les API au curl avec des payloads inventés.** Si tu veux valider qu'un endpoint marche, **utilise l'UI** comme un vrai utilisateur. Les payloads attendus sont définis par les composants frontend qui les construisent — c'est ce comportement qu'il faut tester, pas les coups de curl à la main.

**Règle #2 : avant de déclarer "POST 400 cassé"**, ouvre la page concernée dans Chrome, ouvre DevTools → Network, soumets le formulaire, et capture le payload exact + la réponse exacte. Si l'erreur est `missing field X`, le bug est dans le formulaire frontend, pas dans le backend.

**Règle #3 : avant de déclarer "endpoint absent"**, vérifie dans `backend/src/infrastructure/web/routes.rs` que la route est bien wirée. Cherche le handler concerné dans `backend/src/infrastructure/web/handlers/`. Beaucoup d'endpoints sont nichés sous `/buildings/:id/...` plutôt que des collections plates.

**Règle #4 : seed propre avant de commencer.** Le seed précédent a laissé des données polluées. Re-seed avant ce nouvel audit :
```bash
TOKEN=$(curl -s http://localhost/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@koprogo.com","password":"admin123"}' | jq -r .token)
curl -X POST http://localhost/api/v1/seed/scenario/world \
  -H "Authorization: Bearer $TOKEN"
```
